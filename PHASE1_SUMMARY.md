# Phase 1 多智能体对抗演练系统总结

## 一、阶段定位

Phase 1 的目标不再是军事推演，也不强调特定行业语境，而是做一个最小可运行的对抗演练原型。

本阶段重点是验证三件事：

- 多智能体角色协作链是否能跑通
- LLM 与规则模式是否都能驱动演练
- 终端 / 网页 / 日志三种输出方式是否可用

## 二、当前实现内容

当前系统采用 8x8 网格化对抗场景来表示不同位置，包含：

- 红队 3 个单位：协调单位、侦测单位、执行单位
- 蓝队 2 个单位：应对单位
- 共享信息池：支持目标上报、任务分配、置信度衰减
- 控制台演练入口：`src.battle.runner`
- 网页回放入口：`src.battle.battle_ui`

核心模块仍位于 `src/battle/`，但语义已改为更中性的对抗演练。

## 三、已完成的关键改动

- 将 Phase 1 任务书从特定题材改为“对抗演练”
- 将控制台输出中的行业化术语统一改为对抗态势、单位、位置、对抗、应对等中性表述
- 将 LLM prompt 改为 generic confrontation 语境
- 将红队单位 ID 改为 `red_coordinator`、`red_probe`、`red_striker`
- 将蓝队单位 ID 改为 `blue_guard_1`、`blue_guard_2`
- 将网页回放标题和文案改为对抗演练版本
- 日志目录改为 `outputs/confrontation_logs/`

## 四、当前可用能力

- 可在规则模式下完成一整局对抗演练
- 配置 `SILICONFLOW_API_KEY` 后可启用 LLM 决策
- 侦测单位可发现目标并共享信息
- 协调单位可下发简单任务，执行单位可读取并推进
- 演练过程可输出到控制台、网页和日志文件
- 基于 `unittest` 的最小回归测试已补齐

## 五、已验证结果

已验证以下内容：

- `python3 -m unittest discover -s tests -v` 通过
- 规则模式可运行
- 网页入口可启动
- LLM 模式可正常接入 SiliconFlow 并推进多回合

说明：

- LLM 模式的对局耗时明显高于规则模式，这是外部模型调用的正常结果
- 当前仍然是抽象网格场景，不是更复杂的连续空间或真实业务拓扑

## 六、当前不足

高优先级问题：

- 场景仍然是抽象网格，不是更真实的复杂地图
- 协同还较轻量，尚未形成完整的任务闭环
- LLM 行为稳定性还需要更多回放和统计验证

中优先级问题：

- 胜负判定仍偏粗糙，主要依据单位是否失效和总体状态值
- 日志已落盘，但尚无结构化回放分析工具
- 网页版目前更偏展示，不是交互式演练控制台

低优先级问题：

- 代码目录仍沿用 `battle` 命名，尚未整体重构为更贴近“对抗演练”的模块名
- 部分内部变量名仍保留旧语义，后续可以继续清理

## 七、下一阶段建议

1. 把网格化场景升级为更丰富的地图、区域和目标点
2. 增加地形、补给、控制区、事件卡等更有区分度的对抗元素
3. 完善任务分配闭环，支持更明确的协调 -> 侦测 -> 执行 -> 应对反馈
4. 增加更多自动化测试和回放分析工具
5. 为网页增加单步执行、暂停、筛选日志等功能

## 八、运行方式

控制台规则模式：

```bash
cd /root/autogen-research
python3 -m src.battle.runner
```

控制台 LLM 模式：

```bash
cd /root/autogen-research
./venv/bin/python -m src.battle.runner
```

网页回放：

```bash
cd /root/autogen-research
./venv/bin/python -m src.battle.battle_ui
```
